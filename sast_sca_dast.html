SAST · SCA · DAST

How each tool works, stage by stage — with real examples

SAST — Static Application Security Testing

Analyzes source code without running the app. Reads your code like a security-aware compiler — looking for dangerous patterns, vulnerable functions, and missing security controls. Runs at commit or build time. No app needs to be running.

How SAST works — stage by stage

SAST tools parse your code into an Abstract Syntax Tree (AST), then apply security rules to find dangerous patterns.

1
Code is parsed into an AST (Abstract Syntax Tree)

The tool reads your source code and converts it into a tree structure it can analyze. Every function call, variable assignment, and condition becomes a node in the tree. This is language-specific — Python, Java, JS all have different parsers.

2
Data flow analysis — taint tracking

The tool traces where user input enters the app (called a source) and where it flows to dangerous functions (called a sink). If input flows from a web request directly into a database query without sanitization — that's a SQL injection finding.

3
Rules / patterns are matched

Security rules fire when dangerous patterns are found. Example rule: "if user input reaches os.system() without sanitization, flag command injection." Tools like Semgrep let you write custom rules in YAML for your specific codebase.

4
Findings are reported with file + line number

Output includes the exact file, line, vulnerable code snippet, severity, and remediation guidance. Integrates directly into GitHub PRs so developers see the issue inline before merging.

Vulnerable Python — SAST would flag this
# SAST finding: SQL Injection — user input flows into query unsanitized
def get_experiment(experiment_id):
    # SOURCE: user-controlled input from HTTP request
    query = "SELECT * FROM experiments WHERE id = " + experiment_id
    #                                          ↑ SINK: dangerous — direct string concat
    return db.execute(query)   # ← SAST flags this line: CWE-89

# ─────────────────────────────────────────
# FIXED — parameterized query
def get_experiment(experiment_id):
    query = "SELECT * FROM experiments WHERE id = ?"
    return db.execute(query, [experiment_id])  # ✓ safe
Semgrep custom rule — detect SQL injection pattern
rules:
  - id: sql-injection-string-concat
    patterns:
      - pattern: db.execute(... + $USER_INPUT)
    message: "SQL injection: user input concatenated into query"
    severity: ERROR
    languages: [python]
    metadata:
      cwe: CWE-89
      owasp: A03:2021

What SAST finds well

  • SQL / command / LDAP injection
  • Hardcoded secrets and API keys
  • Weak cryptography (MD5, DES, ECB mode)
  • Missing input validation
  • Insecure deserialization
  • Path traversal vulnerabilities
  • XSS in server-side templates

SAST limitations

  • High false positive rate — needs tuning
  • Cannot find runtime / logic flaws
  • Cannot find auth bypass (needs running app)
  • Cannot find IDOR (business logic)
  • Language-specific — different tool per stack
  • Misses vulnerabilities in 3rd party libs

SCA — Software Composition Analysis

Analyzes your open source dependencies for known vulnerabilities. Modern apps are 80%+ open source code. SCA inventories every library you import, checks it against CVE databases (NVD, OSV, GitHub Advisory), and alerts when a dependency has a known exploit. Does not read your code — reads your dependency files.

How SCA works — stage by stage

SCA builds a Software Bill of Materials (SBOM) then cross-references every component against vulnerability databases.

1
Dependency manifest is parsed

SCA reads your requirements.txt, package.json, pom.xml, or go.sum. It builds a full dependency tree — including transitive dependencies (libraries that your libraries depend on). This is where most vulns hide.

2
SBOM is generated

An SBOM (Software Bill of Materials) is created — a complete inventory of every component, version, and license. This is what you mentioned on your resume at Aptiv. SBOM is now a compliance requirement (US Executive Order 14028).

3
Each component is checked against CVE databases

Every library + version is cross-referenced against NVD (National Vulnerability Database), GitHub Advisory Database, and OSV. If requests==2.19.0 has a known CVE — you get alerted with CVSS score and exploit details.

4
License compliance is checked

SCA also flags license violations — if you're using a GPL library in a commercial product, that's a legal risk. Often overlooked but important in enterprise security programs.

5
Fix version is recommended

The tool tells you exactly which version to upgrade to. Integrated into CI — a PR adding a vulnerable library gets blocked before it merges.

Example — vulnerable requirements.txt (Python)
# requirements.txt
flask==1.0.2          # ← CVE-2018-1000656: DoS via malformed JSON (CVSS 7.5)
requests==2.19.0      # ← CVE-2018-18074: credentials leak via redirect (CVSS 9.8)
pyyaml==3.12          # ← CVE-2017-18342: arbitrary code execution (CVSS 9.8)
django==2.0.0         # ← multiple CVEs — SQL injection, open redirect
cryptography==2.1.4   # ← CVE-2018-10903: weak key generation

# After SCA scan — fixed versions:
flask==2.3.3          # ✓ patched
requests==2.31.0      # ✓ patched
pyyaml==6.0.1         # ✓ patched

What SCA finds

  • Known CVEs in direct dependencies
  • Known CVEs in transitive dependencies
  • License violations (GPL, AGPL conflicts)
  • Outdated / unmaintained packages
  • Malicious packages (typosquatting)
  • Supply chain attacks (Log4Shell style)

SCA limitations

  • Only finds known CVEs — not zero days
  • Cannot find custom code vulnerabilities
  • Noisy — many low-severity findings
  • False positives if vuln code path not used
  • Reachability analysis needed to reduce noise

DAST — Dynamic Application Security Testing

Tests the running application from the outside — exactly like an attacker would. Sends real HTTP requests with malicious payloads and observes the response. Finds runtime vulnerabilities that SAST and SCA cannot see. App must be running — DAST tests behavior, not code.

How DAST works — stage by stage

DAST acts as an automated attacker — crawling, probing, and attacking a live app to find exploitable vulnerabilities.

1
Target is configured and authenticated

You point DAST at a staging URL. You provide auth credentials or a session token so it can test authenticated endpoints. Without auth, DAST only tests public-facing pages — missing 80% of the attack surface.

2
Spider / crawl discovers all endpoints

DAST crawls the app — following links, submitting forms, calling API endpoints — to build a map of everything it can test. For SPAs (React/Angular), you may need to provide an OpenAPI spec or record a browser session.

3
Active scanning — attack payloads are injected

For every parameter on every endpoint, DAST injects attack payloads — SQL injection strings, XSS vectors, path traversal sequences, SSRF URLs. It fires thousands of requests automatically.

4
Responses are analyzed for vulnerability indicators

DAST looks for signs of exploitation in responses: SQL error messages, reflected script execution, unexpected data in responses, different response times (for blind injection). If ' causes a 500 error with a DB stack trace — SQL injection confirmed.

5
Findings reported with request / response evidence

Each finding includes the exact HTTP request sent, the response that triggered the alert, and the specific payload used. This makes it easy for developers to reproduce and fix the issue.

What DAST actually sends — XSS probe example
# DAST sends this request to every input parameter:
GET /api/search?q=<script>alert(document.cookie)</script> HTTP/1.1
Host: app.benchling.com
Cookie: session=eyJhbGciOiJIUzI1NiJ9...

# Vulnerable response — script tag reflected back:
HTTP/1.1 200 OK
Content-Type: text/html
<h2>Results for: <script>alert(document.cookie)</script></h2>
# ↑ DAST flags: Reflected XSS — input returned in HTML without encoding

# Safe response — input properly encoded:
HTTP/1.1 200 OK
<h2>Results for: &lt;script&gt;alert(1)&lt;/script&gt;</h2>
# ↑ HTML entities — XSS neutralized
DAST in CI pipeline — OWASP ZAP GitHub Action
name: DAST Security Scan
on: [push]
jobs:
  zap-scan:
    runs-on: ubuntu-latest
    steps:
      - name: Deploy staging app
        run: docker-compose up -d

      - name: Run OWASP ZAP DAST scan
        uses: zaproxy/action-full-scan@v0.9.0
        with:
          target: 'http://localhost:8080'
          rules_file_name: '.zap/rules.tsv'
          cmd_options: '-a'   # authenticated scan

      - name: Fail pipeline on High findings
        run: |
          if grep -q "HIGH" zap-report.json; then
            echo "High severity findings — blocking merge"
            exit 1
          fi

What DAST finds (SAST cannot)

  • Runtime injection (SQLi, XSS, CMDi)
  • Authentication / session management flaws
  • CORS misconfigurations
  • Security headers missing (CSP, HSTS)
  • SSRF — server-side request forgery
  • API rate limiting failures
  • Business logic flaws (partially)

DAST limitations

  • Needs running app — can't run on code alone
  • Slow — full scan can take hours
  • Misses IDOR (can't know valid IDs to test)
  • Misses logic flaws requiring human judgment
  • Can break staging environments if not careful
  • Auth setup complex for modern SPAs

All three tools together — the DevSecOps pipeline

SAST, SCA, and DAST each catch different things. None of them alone is enough. Used together in a CI/CD pipeline, they create overlapping layers of protection — catching issues at code, dependency, and runtime level before anything reaches production.

The key insight — what each tool misses

  • SAST + SCA together still cannot find IDOR, auth bypass, or logic flaws
  • DAST alone misses hardcoded secrets and vulnerable dependencies
  • All three together still miss business logic flaws — that needs manual pentesting
  • The full picture: SAST + SCA in CI, DAST on staging, manual pentest before major releases
